# fireworks-cli

協力型カードゲームを、コマンドラインから操作できるように実装したツールです。ゲームの進行状態は 1 ゲームにつき 1 つの YAML ファイルに記録され、手の履歴はすべてそのファイルに追記されていきます。ファイルを書き換えるのではなく追記していく方式のため、過去の手を遡って参照したり、同じゲームを再生したりすることができます。

プレイヤーごとに「自分の手札が見えない」状態を再現するため、表示時にフィルタをかける仕組みを持ちます。ファイル自体には全プレイヤーの手札が保存されていますが、コマンド実行時に自分を指定すると、自分の手札だけが伏せられた状態で画面に出力されます。

**ゲームルールは「ハナビ（Hanabi）」** に基づいています。Hanabi は Antoine Bauza 氏が設計し、Cocktail Games（フランス）から 2010 年に発売された作品です。本 CLI は、**ルール体系のみを独自に再実装した非営利のオープンソース** です。公式ゲームのイラスト、ルールブックの本文、その他著作物は一切含んでいません。詳細は末尾の [**クレジットと商標表示**](#クレジットと商標表示) を参照してください。

このツールは、既存の CLI [`eris-ths/guild-cli`](https://github.com/eris-ths/guild-cli) v0.3.0 から派生したもので、ファイル操作、並行書き込み対策、入力検証、アーキテクチャ構成はそのまま引き継いでいます。ゲーム固有のルール部分（カードの定義、ターン進行、勝敗判定など）は新規に実装しました。

> 現状はアルファ版（0.x 系列）です。脅威モデルは [`SECURITY.md`](./SECURITY.md)、変更履歴は [`CHANGELOG.md`](./CHANGELOG.md) にあります。
> 英語版は [`README.md`](./README.md) にあります。

## このツールを作った目的

3 つの目的があります。

1. **既存 CLI の設計思想（guild-cli）がゲーム領域でも通用するかの検証**。 ファイル追記型の記録方式、並行書き込みの安全性、視点フィルタの仕組みが、チーム協働ツールで有効に機能することは確認済みでした。同じ考え方が、ターン制の協力カードゲームにも応用できるかを確かめる目的があります。
2. **視点フィルタ機能の実用検証**。ハナビは「自分の手札だけ見えない」というルールを持つゲームのため、視点切り替え機能を試す題材として適しています。データの保存形式は 1 つのまま、表示時に視点を切り替える設計が実用に耐えるかを確認します。
3. **後継プロジェクト「atelier」への布石**。汎用的な AI ファーストのゲーム制作基盤を設計中で、その一段目のプロトタイプとして本プロジェクトを位置づけています（[`docs/ATELIER.md`](./docs/ATELIER.md) 参照）。

## ゲームそのものの評価

ハナビ（Antoine Bauza 氏設計）は、ドイツ語圏のボードゲーム最高賞である **Spiel des Jahres** を 2013 年に受賞した作品で、協力型カードゲームの古典として高く評価されています。以下の特徴があります。

- **自分の手札だけが見えない**。他のプレイヤーの手札は見える。自分が出せるヒントは、自分では受け取れない。この非対称性がゲームの中心構造です。
- **情報トークンが有限（初期 8 個）**。ヒントを出すごとに 1 個消費し、カードを捨てると 1 個回復します。コミュニケーション資源が常に逼迫しています。
- **ミスは 3 回で敗北**。1 ゲームを通して緊張状態が持続します。
- **相手を責める構造にならない**。協力ゲームのため、ミスが起きたときの話題は「どうすれば情報を伝えられたか」になります。
- **1 ゲームの所要時間は 15〜30 分**。

本 CLI でプレイする場合、物理版にはない以下の特性があります。

- **非同期プレイが容易**。ゲーム状態は 1 つの YAML ファイルに集約されるため、ファイルを共有すれば、離れた場所にいるプレイヤーとも時間を空けてプレイできます。
- **ゲームの完全再生が可能**。シード値と手の履歴から任意のターンを再現できます。
- **AI プレイヤーとの対戦を想定した拡張余地**。観戦 AI、協力 AI の追加は `atelier` 側の設計に組み込まれています。

ハナビが未経験の方には、本 CLI の 2 人用設定が入門として適しています。

## 動作環境とインストール

Node.js 20 以降が必要です。

```bash
npm install
npm run build
node ./bin/fireworks.mjs --help
```

## 基本的な使い方

```bash
# 新しいゲームを開始
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs new-game --players eris,nao --seed "my-seed"

# 盤面を表示（観戦者視点。全プレイヤーの手札が見える）
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs show

# 盤面を表示（特定プレイヤー視点。そのプレイヤーの手札だけ伏せる）
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs show --as eris

# 手札の 2 枚目を場に出す
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs play 2 --by eris

# 手札の 3 枚目を捨てる
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs discard 3 --by eris

# 相手にヒントを出す（数字ヒント、または色ヒント）
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs inform --by eris --target nao --rank 1
FIREWORKS_ROOT=./demo node ./bin/fireworks.mjs inform --by eris --target nao --color blue
```

## 提供コマンド一覧

| コマンド | 役割 | 主な引数 |
|---|---|---|
| `new-game` | 新しいゲームを作成する | `--players`（参加プレイヤー、カンマ区切り）, `--seed`（シード値）, `--id`（ゲーム識別子） |
| `show` | 現在の盤面を表示する | `--as`（視点指定）, `--id` |
| `play` | 手札からカードを場に出す | 手札位置（1 から始まる番号）, `--by`（実行プレイヤー） |
| `discard` | 手札からカードを捨てる | 手札位置, `--by` |
| `inform` | 相手にヒントを出す | `--by`, `--target`, `--color` または `--rank` |
| `list` | 保存されているゲーム ID 一覧を表示する | — |
| `help` | 使い方を表示する | — |

すべてのコマンドで、想定外の引数は即座にエラーとして検出されます（入力ミスを黙って無視しない方針）。詳細は [`AGENT.md`](./AGENT.md) を参照してください。

## 設計の要点

guild-cli から引き継いだ設計を、ゲーム領域に合わせて適用しています。

- **ゲームの進行履歴を真実の源とする**。現在のゲーム状態は、履歴から導出できる派生データとして扱います。これにより、ゲームの再生、過去の手の振り返り、任意時点の状態復元が可能になります。
- **状態変更は新しい状態を返す純粋な関数として実装**。元の状態を直接書き換えません。保存は、ファイルを一時ファイルに書いてからリネームする方式で、途中状態が他プロセスに見えないようにします。
- **並行書き込みの検出**。保存時にファイル中のバージョン番号を確認し、読み込み時と変わっていれば書き込みを中止してエラーとします（同時に同じゲームを操作した場合の衝突検出）。
- **視点フィルタは表示段階で適用**。ファイルには全情報を保持し、画面表示のときだけフィルタをかけます。
- **乱数は決定的**。シード値を固定すれば、シャッフルの結果は完全に再現されます。エンジン内で JavaScript 標準の乱数関数（`Math.random()`）は一切使っていません。
- **引数検証は厳格**。コマンドごとに受け付ける引数を明示的に定義し、それ以外はエラーとします。

## 実装済みのルール（標準ハナビに準拠）

- カード 50 枚（5 色 × 各色 10 枚構成：1 が 3 枚、2〜4 が各 2 枚、5 が 1 枚）
- 手札枚数：2〜3 人プレイで 5 枚、4〜5 人プレイで 4 枚
- 情報トークン 8 個（ヒントで 1 個消費、カードを捨てると 1 個回復、5 を場に出すと 1 個回復。上限 8）
- ミス 3 回で敗北
- デッキ枯渇後は残プレイヤーが 1 手ずつプレイしてゲーム終了
- 25 点（全色の花火を 1〜5 まで完成させる）で勝利

## セキュリティと品質の方針

- 脅威モデルは [`SECURITY.md`](./SECURITY.md) に記載。
- TypeScript の厳格モード（`strict`、`noUncheckedIndexedAccess`、`exactOptionalPropertyTypes`）を有効化。型安全性を最大限確保しています。
- すべてのコマンドに動作確認テストを用意しています。
- Lua サンドボックス（後述）の禁止項目はテストで固定されています。
- エンジン内では `Math.random()` を使わず、すべての乱数をシード管理下に置いています。これにより全ゲームが監査可能・再現可能です。

## Lua サンドボックスについて（実験機能）

`src/core/lua/sandbox.ts` に、wasmoon ライブラリをベースにした Lua 実行環境があります。以下の Lua 標準ライブラリは、サンドボックス起動時に無効化されます：`os`, `io`, `require`, `package`, `debug`, `load`, `loadstring`, `loadfile`, `dofile`, `collectgarbage`。ファイルシステムアクセスや外部コード読み込みができない状態になります。

純粋計算に必要なモジュール（`math`, `string`, `table`）は利用可能です。

現状、ハナビ本体のルール処理には Lua は組み込まれていません。本サンドボックスは、後継プロジェクト `atelier` のカード定義スクリプト基盤として用意されたものです。

- 機密情報はコミットしません（環境変数ファイルも想定外）。

## クレジットと商標表示

- **ゲームデザイン**: ハナビ（Hanabi）は **Antoine Bauza**（フランス）が設計し、**Cocktail Games** から 2010 年に初版が発売された作品です。2013 年に **Spiel des Jahres**（ドイツ語圏のボードゲーム最高賞）を受賞しました。その後、R&R Games（北米）、Abacusspiele（ドイツ）などから各国語版が発売されています。
- 本 CLI は、イベントソーシング型の記録方式が隠し情報のあるゲームにも適用できるかを検証する目的で作られた、**独立した非営利のオープンソース再実装** です。Antoine Bauza 氏、Cocktail Games、R&R Games、Abacusspiele のいずれとも関連・提携・後援関係はありません。

- 「Hanabi」 は関係各社の商標である可能性があります。本リポジトリでの名称使用は、どのゲームのルールを実装しているかを明示する目的の範囲に留めており、関係性や後援を示唆するものではありません。
- ゲームを気に入った方は、**公式の物理版を正規の販売店から購入すること** を強く推奨します。デザイナーと出版社を支援することになります。
